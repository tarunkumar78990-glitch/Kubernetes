# platform-ops

The SRE layer: namespaces, quotas, default-deny networking, monitoring, SLOs,
burn-rate alerts, and runbooks.

**This is not an application repo.** It's the foundation everything else lands on.

---

## Apply it first

Order is not cosmetic.

```bash
gcloud container clusters get-credentials dev-gke \
  --region asia-south1 --project YOUR_PROJECT

./scripts/bootstrap.sh
```

That runs, in order:

1. **Namespaces** — `dev`, `staging`, `prod`, `monitoring`, with Pod Security
   Admission set to `restricted`
2. **Quotas + LimitRanges** — all three environments share one 2-node cluster, so
   without these a runaway dev HPA can starve prod
3. **Default-deny NetworkPolicies** — *before* workloads exist
4. **Monitoring** — Prometheus, Alertmanager, Grafana, and the SLO rules

### Why default-deny goes first

Apply it *after* services are running and you get a window where everything can
reach everything — and a confusing afternoon when you finally close it. Apply it
first and every service's own NetworkPolicy opens exactly the paths it needs.

Without default-deny, the per-service policies in each repo are **decorative**.
With it, a compromised `recommendation` pod cannot reach `payment` at all.

## SLOs

`slo/recording-rules.yaml` and `slo/burn-rate-alerts.yaml`.

| Service | Tier | Availability | Latency | Budget / 30d |
|---|---|---|---|---|
| checkout | 1 | 99.95% | 800ms | 21.6 min |
| payment | 1 | 99.95% | 500ms | 21.6 min |
| user-auth | 1 | 99.95% | 400ms | 21.6 min |
| frontend | 1 | 99.9% | 500ms | 43.2 min |
| order | 1 | 99.9% | 400ms | 43.2 min |
| cart | 2 | 99.9% | 300ms | 43.2 min |
| product-catalog | 2 | 99.9% | 300ms | 43.2 min |
| shipping | 2 | 99.5% | 500ms | 3.6 hrs |
| notification | 3 | 99.0% | 1000ms | 7.2 hrs |
| recommendation | 3 | 99.0% | 400ms | 7.2 hrs |

Note that `notification` and `recommendation` have deliberately loose SLOs. They
are best-effort by design — checkout does not fail if the email doesn't send.
Giving them 99.95% would mean paging someone at 3am for an email retry.

### Burn-rate alerting, and why not "error rate > 0"

A 99.9% target allows 43 minutes of downtime per 30 days. Alert on raw errors and
you page for a blip that costs 4 seconds of budget. Alert on **burn rate** and
you page only when the incident will genuinely exhaust the budget.

Four alerts per service:

| Alert | Burn | Windows | Action |
|---|---|---|---|
| FastBurn | 14.4x | 5m **and** 1h | **Page.** Budget gone in ~2 days. |
| SlowBurn | 6x | 30m **and** 6h | Ticket. Real, not urgent. |
| LatencySLOBreach | — | 1h | Ticket. |
| BudgetExhausted | — | 30d | Page + **freeze releases**. |

The `and` between two windows is the whole trick. The short window makes it fast;
the long window stops a 30-second blip from paging you. Either window alone gives
you a choice between slow alerts and a noisy pager.

## Alert routing

`monitoring/alertmanager.yaml`:

- `severity: critical` → PagerDuty. Wakes someone.
- `severity: warning` → Slack ticket channel. **Never pages.**
- `namespace: dev|staging` → a noise channel. Never pages anyone, ever.

Inhibit rules stop cascades: if a node is down you get one page, not thirty.
If the budget is exhausted, the fast-burn alert is suppressed as redundant.

**Before this works:** replace `REPLACE_WITH_SLACK_WEBHOOK` and
`REPLACE_WITH_PAGERDUTY_ROUTING_KEY`.

## Runbooks

One per service in `runbooks/`. Each burn-rate alert carries a `runbook_url`
annotation pointing at it, so the page links to the fix.

Every runbook opens with the same three commands, because the answer is nearly
always the third one:

```bash
kubectl get pods -n prod -l app=<service> -o wide     # one pod or all?
kubectl rollout history deployment/<service> -n prod  # what changed?
kubectl logs -n prod -l app=<service> --tail=100      # what does it say?
```

If a deploy went out in the last hour: **roll back first, investigate second.**

```bash
kubectl port-forward -n monitoring svc/prometheus 9090:9090
kubectl port-forward -n monitoring svc/grafana 3000:3000
kubectl port-forward -n monitoring svc/alertmanager 9093:9093
```

Grafana's admin password is generated by `bootstrap.sh` and printed once. Move it
into Secret Manager.

## Expected on day one

The SLO rules reference metrics your services emit. Until you deploy a service
**and send it traffic**, the rules evaluate to `NaN` and the alerts stay silent.

That is correct behaviour, not a broken install. Generate some traffic and the
dashboards populate.
